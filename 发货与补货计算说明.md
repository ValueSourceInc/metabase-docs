# 发货与补货计算说明（大白话版）

---

## 一、先理清数据链路：一张图看懂谁喂谁

```
#877 发货基础表  ──┐
#876 销量预测(按天) ─┤
#899/#900 在途货件  ─┤
                   ▼
        #878 备货发货 Forecast WOS Summary   ← 真正干活的"发动机"（原生SQL）
                   │
                   ▼
        #873 补货 FBA Total Inventory Chart 2.0   ← 878 套了一层壳，字段全来自878
                   │
                   ▼
        #901 工厂补货表2.0   ← 在873基础上，按SKU汇总US/CA，再算"要生产多少"
```

一句话：**878 算"要发多少货"（need_to_ship、allocated_qty），901 在它之上算"要生产多少"（total_need_to_produce_with_moq）。** 873 只是把 878 的结果摆出来给人看，本身不做计算。

## 二、几个必须先懂的核心概念

| 概念                       | 字段                             | 大白话                                                                                                      |
| -------------------------- | -------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| **WOS（Weeks of Supply）** | `wos_now`                        | 当前库存还能卖多少周。断货天数 ÷ 7。算不出来就兜底 100 周（≈2年，因为预测只覆盖未来 700 天）。              |
| **断货天数**               | `stockout_days`                  | 按预测销量一天天扣库存，扣到不够卖的那一天，距今还有几天。                                                  |
| **在途异常库存**           | `abnormal_inventory_*`           | 在路上但状态不对（delayed 延误 / lost 丢失 / return_ship 退回）的货。这部分**不算可用库存**，因为指望不上。 |
| **退货在途量**             | `return_ship_quantity`           | 状态是 return_ship 的那部分，会退回来，**未来能重新卖**，所以单独拎出来。                                   |
| **安全库存周数**           | `safety_wos`（默认 30/7≈4.3 周） | 到货后想至少再撑这么久的安全库存。`safety_days` = safety_wos × 7。                                          |
| **发货在途天数**           | `shipping_days`（默认 30 天）    | 这次发货从工厂到 FBA 大约要多少天。                                                                         |
| **整箱规格**               | `casepack`                       | 一个整箱装几件。发货/生产都要凑整箱。                                                                       |

## 三、#873：要发多少货？

### 1. `need_to_ship` —— "理论上该发多少"

**核心思路：算到货那一刻会不会有缺口。**

按这个顺序推演（针对每个 SKU × 店铺 × 站点 US/CA）：

1. **今天有多少可用库存** = 当前库存 − 在途异常库存（延误/丢失/退回的那批指望不上）。
2. **发货在路上这几天还会卖掉多少** = `demand_until_arrival`（从今天到到货日，这 shipping_days 天的预测销量之和）。
3. **到货时预计还剩多少库存** = 当前库存 − 路上卖掉的 − 到货时仍是异常的库存。
4. **到货后想再撑 safety_days 天（安全库存期）**，这期间还会卖掉一批。
5. **缺口 = 到货后安全库存期内，累计需求 − 当时可用库存 的最大值**（`max_arrival_safety_deficit`）。

```
need_to_ship = 向上取整( max(0, 到货后安全库存期的缺口) )
```

> **为什么这么做？** 不是简单看"现在缺多少"，而是看"货到了之后、再过完安全库存期，会不会还是不够"。这样发出去的量，能保证货到岸后还能安稳撑过安全库存期，不会刚到货又快断货。

`need_to_ship_with_casepack` 就是把上面这个数再**向上凑成整箱的倍数**（casepack）。

---

### 2. `allocated_qty` —— "结合实际情况，真实准备发的量"

`need_to_ship` 是"按需全给"，但现实中**每个 parent_asin × 每个站点 一票最多发 500 件**（`shipment_cap` 默认 500），装不下就得排队。`allocated_qty` 就是在这个上限内分配出来的真实发货量。

> **分组维度**：按 `parent_asin_code × Marketplace` 分组。例如 parent_asin = "DH-BT-BRAID"，则 US 站点单独一批 500、CA 站点单独一批 500，互不挤占。查不到 parent_asin_code 的 SKU 归到 NULL 组，也独立分一批。

**分配规则：两阶段，按"谁快断货谁优先"排队。**

**阶段一（救急 rescue）：** 把库存覆盖低于"安全库存期 × `safety_wos_ratio`"的 SKU，先补到这个目标。

- 救急目标 = `safety_days × safety_wos_ratio`（`safety_wos_ratio` 默认 **1.0**，即补到完整安全库存期）。
- 按 `wos_now` 从小到大排（越快断货越靠前）。
- 每个 parent×站点 500 件的额度，前面 SKU 用掉多少，后面接着用剩下的。
- 排到额度用完，后面的这一阶段就拿不到了。

**阶段二（贪心补足）：** 救急后剩下的额度，继续按"谁快断货谁优先"，把还没满足的 `need_to_ship` 尽量补上。

- 还是按 `wos_now` 排序，一个个贪心给，直到 500 件额度用完。

**最后凑整箱：**

```
allocated_qty = 向下取整到整箱倍数 (阶段一 + 阶段二 的量)
```

> **关键约束：`allocated_qty` 永远 ≤ `need_to_ship_with_casepack`**。每个 SKU 分到的量不会超过它自己的需求。
>
> **为什么这么做？**
>
> - **按 parent×站点 设 500 件上限**：物流按 parent 发货，每个 parent 的 US/CA 各发一票，互不影响。
> - **先救急再补足**：最快断货的优先保命，剩下的额度再按紧迫度补，避免"额度被慢断货的吃掉、快断货的反而没发"。
> - **救急目标用 safety_wos 的比例**：`safety_wos_ratio` 是 0~1 的小数，表示救急要补到 safety_wos 的百分之多少。永远 ≤ safety_wos，不会出现"救急量超过安全库存"的无解情况。默认 1.0 = 补到完整安全库存期（等价于全量按需分配）；调小（如 0.5）= 救急只补一半，留更多额度给阶段二按 wos 紧迫度排队分配。
> - **向下凑整箱**：物流按箱走，零头发了麻烦，宁可少发几件也要整箱。
> - 两个参数可调：`shipment_cap`（每 parent×站点上限）、`safety_wos_ratio`（救急目标占 safety_wos 的比例），改这两个值就能调分配策略。

## 四、#901：要生产多少？

901 把 873 按 SKU 汇总，把 US、CA 两个站点的需求并排，再扣掉工厂已有的库存，最后算出"工厂要生产多少"。

> **字段取值约定**：`active_product_num_days`（生产周期）、工厂库存 `stock`、`moq` 这三个量都是 **US 优先、US 为 0 才取 CA**（`CASE WHEN us≠0 THEN us WHEN ca≠0 THEN ca ELSE 0`）——同一个 SKU 的 US/CA 共享一套配置，优先用 US 的；而 `need_to_ship`、日销则各站点独立。

### 1. `us_need_to_produce` / `ca_need_to_produce` —— "每个站点要生产多少"

```
需要生产量 = 需要发货量(need_to_ship)
           + (生产周期 + 工厂安全库存期的一半) × 日销量
```

精确点：

```
us_need_to_produce = 向上取整(
    us_need_to_ship
  + ( active_product_num_days          ← 生产周期(天)
      + factory_safety_wos × 7 ÷ 2     ← 工厂安全库存1周，按半周期 3.5 天 折算
    ) × us_estimated_daily_sales        ← 近期平均日销
)
```

> **为什么这么做？** 发出去的货（`need_to_ship`）只是这一票要补的。但从**下单生产到货发出去**这段生产周期里，库存还在继续卖；同时工厂本身也要留一点安全库存。所以除了"要发的量"，还得把"生产这段时间还会卖掉的量"一起生产出来，否则下次又断货。工厂安全库存按半周期（1 周的一半 ≈ 3.5 天）折算，是补货节奏的约定。

### 2. `total_need_to_produce` —— "工厂总共要生产多少"

US 和 CA 各自的要生产量加起来，再扣掉**工厂现有库存**和**退货在途量**（退回来的货能重新用，省得重新生产）：

```
total_need_to_produce = 向上取整( max(0,
    us_need_to_produce + ca_need_to_produce
    - 工厂现有库存(stock)
    - 退货在途量(return_ship_quantity)
))
```

> **为什么扣这两项？**
>
> - **扣工厂库存**：工厂仓库里还有的货，不用重新生产，直接抵掉。
> - **扣退货在途**：状态是 return_ship 的货会退回来、能再卖，相当于"快到的库存"，提前抵扣，避免重复生产。
> - **max(0, ...)**：扣完若是负数（库存很充足），就归零，不生产。

### 3. `total_need_to_produce_with_moq` —— "最终下单量（考虑最小起订量）"

工厂有**最小起订量 MOQ**（一次最少做多少）。所以最终下单量不是"需要多少就生产多少"，而是按 MOQ 规则取整：

```
如果 total_need_to_produce ≥ MOQ          → 按需生产（需要多少产多少）
如果 total_need_to_produce ≥ MOQ 的 70%    → 按 MOQ 生产（差得不多，凑个起订量）
否则                                        → 不生产（需求太小，不够起订量就不下单）
```

> **为什么要这条 MOQ 规则？** 工厂不愿意为了几十件开机。需求达到 7 成起订量，就值得凑整到 MOQ 下一单；连 7 成都不到，说明这个 SKU 暂时不急，干脆这轮不下单，等下一轮攒够了再说，避免零碎小单。`total_need_to_produce_with_moq` 就是业务最终看的"这单下多少"。

## 五、算一遍：一个 SKU 的完整计算过程

下面用一个虚拟 SKU **"QS001-T-ST-DG"** 把全链路算一遍。数字是编的，只为讲清逻辑，不代表真实数据。

**前提参数（该 SKU 在 US 站点）：**

| 项 | 值 | 说明 |
| --- | --- | --- |
| 当前可用库存 | 205 件 | 今天 FBA 能卖的库存 |
| 在途异常库存（到货时仍异常） | 20 件 | 延误/丢失，指望不上 |
| 预测日销 | 5 件/天 | `estimated_daily_sales_after_shipping_days` |
| 发货在途天数 `shipping_days` | 30 天 | 工厂发到 FBA 要 30 天 |
| 安全库存 `safety_wos` | 30/7 周 → `safety_days`=30 天 | 到货后想再撑 30 天 |
| 整箱规格 `casepack` | 12 件 | 一箱 12 件 |
| `wos_now` | 0.5 周 | 当前只够卖 3.5 天，很急 |

---

### 第一步：算 `need_to_ship`（理论该发多少）

**① 货在路上 30 天，会卖掉多少？**

```
demand_until_arrival = 30 天 × 5 件/天 = 150 件
```

**② 货到岸时，预计还剩多少可用库存？**

```
到货时可用库存 = 当前库存 − 路上卖掉的 − 到货时仍异常的
             = 205 − 150 − 20 = 35 件
```

**③ 到货后再过 30 天安全库存期，这期间还要卖多少？**

```
安全期销量 = 30 天 × 5 件/天 = 150 件
```

**④ 缺口 = 安全期内累计需求 − 到货时可用库存 的最大值**

随着安全期一天天过去，累计需求从 0 涨到 150，可用库存一直是 35（不再补）。缺口最大出现在安全期最后一天：

```
max_arrival_safety_deficit = 150 − 35 = 115 件
```

**⑤ `need_to_ship` = 向上取整**

```
need_to_ship = ceil(max(0, 115)) = 115 件
```

**⑥ `need_to_ship_with_casepack` = 向上凑整箱**

```
ceil(115 ÷ 12) × 12 = ceil(9.58) × 12 = 10 × 12 = 120 件
```

> 到这里：理论上该发 115 件，凑整箱后 120 件。但实际能不能发这么多，看下一步排队。

---

### 第二步：算 `allocated_qty`（实际准备发多少）

`need_to_ship` 是"按需全给"，但同一 parent × US 站点一票最多发 500 件（`shipment_cap`=500），装不下就排队。假设该 parent 的 US 站点这票有 3 个 SKU 要发，按"谁快断货谁优先"（`wos_now` 升序）排：

| 排序 | SKU | wos_now(周) | need_to_ship | rescue_qty(补到 safety_wos×0.5) |
| --- | --- | --- | --- | --- |
| 1 | **QS001（本例）** | 0.5 | 115 | 40 |
| 2 | SKU-B | 1.2 | 200 | 10 |
| 3 | SKU-C | 2.0 | 300 | 0 |

> 这里 `safety_wos_ratio` 取 **0.5** 演示两阶段分配。rescue_qty 是「到货后前半段安全库存期（ratio×safety_days）内的缺口」，比 need_to_ship 小，因为 need 是整个安全库存期（safety_days）的累计缺口（本例 QS001：到货剩 35 件、日销 5 件，前 15 天缺口 5×15−35=40，整段 30 天缺口 5×30−35=115）。**默认 ratio=1.0 时两者相等**（救急窗口=整段安全期），阶段一即覆盖全部 need、阶段二为空；调小 ratio 才留出额度给阶段二按紧迫度排队。

**阶段一 · 救急**（按 wos_now 升序，先补各 SKU 在 ratio×safety_days 窗口内的缺口 rescue_qty，cap=500）：

| SKU | 前面已用 | 本轮拿到 | 说明 |
| --- | --- | --- | --- |
| QS001 | 0 | min(40, 500−0) = **40** | 救急拉到 1 周 |
| SKU-B | 40 | min(10, 500−40) = **10** | |
| SKU-C | 50 | 0 | 已≥1周，不用救急 |

阶段一合计用掉 50 件，剩余额度 450 件。

**阶段二 · 贪心补足**（剩余额度按紧迫度继续补 `need_to_ship`）：

| SKU | 剩余需求 (need−phase1) | 前面已用 | 本轮拿到 |
| --- | --- | --- | --- |
| QS001 | 115−40 = 75 | 0 | min(75, 450−0) = **75** |
| SKU-B | 200−10 = 190 | 75 | min(190, 450−75=375) = **190** |
| SKU-C | 300−0 = 300 | 265 | min(300, 450−265=185) = **185** ← 额度用完，被截断 |

阶段二合计用掉 450 件，额度刚好用尽。

**合计（凑整箱前）**：QS001 = 40+75 = 115；SKU-B = 10+190 = 200；SKU-C = 0+185 = 185。

**最后向下凑整箱**（casepack=12）：

| SKU | 合计 | 向下凑整箱 | `allocated_qty` |
| --- | --- | --- | --- |
| QS001 | 115 | floor(115÷12)×12 = 9×12 = 108 | **108** |
| SKU-B | 200 | floor(200÷12)×12 = 16×12 = 192 | **192** |
| SKU-C | 185 | floor(185÷12)×12 = 15×12 = 180 | **180** |

> 本例 QS001：理论要发 115，但凑整箱丢了 7 件零头，**实际这票发 108 件**。SKU-C 则因为额度用完被截断（要 300 只发 185）。
>
> ⚠️ **注意一个关键点**：下一步算"要生产多少"用的是 `need_to_ship`（理论值 115），**不是** `allocated_qty`（实际发的 108）。也就是说，即使这票发货被 500 件上限卡住，生产量仍按"理论该发多少"来备——因为生产是为补货兜底，不是只补这一票。

---

### 第三步：算 `us_need_to_produce`（US 站点要生产多少）

```
us_need_to_produce = ceil( us_need_to_ship + (生产周期 + 工厂安全库存半周期) × 日销 )
```

代入：

| 项 | 值 |
| --- | --- |
| us_need_to_ship | 115 件（理论发货量，不是 108） |
| 生产周期 `active_product_num_days` | 45 天 |
| 工厂安全库存 `factory_safety_wos`=1 周 → 1×7÷2 | 3.5 天（按半周期折算） |
| US 日销 | 5 件/天 |

```
= ceil( 115 + (45 + 3.5) × 5 )
= ceil( 115 + 48.5 × 5 )
= ceil( 115 + 242.5 )
= 358 件
```

> 比"要发的 115"多出来约 243 件，是因为从下单生产到货发出去这 48.5 天（45+3.5）里，库存还在以每天 5 件的速度卖，这部分也要一起生产出来，否则下次又断货。

---

### 第四步：算 `total_need_to_produce`（工厂总共要生产多少）

把 US 和 CA 两个站点的要生产量加起来，再扣工厂现有库存和退货在途。

假设 CA 站点该 SKU：`ca_need_to_ship`=50、CA 日销=3、同样生产周期 45 天：

```
ca_need_to_produce = ceil( 50 + (45+3.5) × 3 ) = ceil( 50 + 145.5 ) = 196 件
```

再假设：工厂现有库存 300 件，退货在途 20 件。

```
total_need_to_produce = ceil( max(0,
    us_need_to_produce + ca_need_to_produce − 工厂库存 − 退货在途
))
= ceil( max(0, 358 + 196 − 300 − 20) )
= ceil( 234 )
= 234 件
```

> 工厂仓里已有 300 件、还有 20 件退货要回来，能抵掉 320 件，所以净生产需求从 554 降到 234。

---

### 第五步：算 `total_need_to_produce_with_moq`（最终下单量）

设该 SKU 的 MOQ = 500 件。按 MOQ 规则判定：

```
234 ≥ 500 ?         否
234 ≥ 500 × 70% (=350) ?   否
→ 不下单，total_need_to_produce_with_moq = 0
```

本例需求 234 件，连 7 成起订量（350）都不到，这轮不下单，等下一轮攒够了再说。

**三种情况对比（同样 MOQ=500）：**

| SKU | total_need_to_produce | 判定 | 最终下单量 |
| --- | --- | --- | --- |
| QS001（本例） | 234 | < 350（7成） | **0**（不下单） |
| SKU-D | 380 | 350 ≤ 380 < 500 | **500**（凑 MOQ） |
| SKU-E | 620 | ≥ 500 | **620**（按需生产） |

---

### 全链路一图回顾（QS001）

```
当前库存205 → 到货时剩35 → 安全期缺口115
  → need_to_ship = 115           (理论该发)
  → 排队+凑整箱 → allocated_qty = 108   (这票实际发)
  → us_need_to_produce = 358      (加上生产周期内的销量)
  → +CA(196) −工厂库存(300) −退货(20)
  → total_need_to_produce = 234   (工厂净生产需求)
  → MOQ判定(234<350) → with_moq = 0    (这轮不下单)
```

## 六、可调参数一览（业务可以动这几个旋钮）

| 参数                 | 在哪     | 默认      | 作用                                                                              |
| -------------------- | -------- | --------- | --------------------------------------------------------------------------------- |
| `shipment_cap`       | #878     | 500       | 每个 parent_asin × 站点 一票最多发多少件。调大→一次发更多、覆盖更全；调小→更保守、分多票。 |
| `safety_wos_ratio`   | #878     | 1.0       | 救急目标占 safety_wos 的比例（0~1 小数）。1.0=补到完整安全库存期；调小（如0.5）=救急只补一半，留额度给阶段二按紧迫度排队。 |
| `safety_wos`         | #877→878 | ~4.3 周   | 到货后想留的安全库存周数。调大→need_to_ship 偏大、更保险。                        |
| `shipping_days`      | #877→878 | 30 天     | 发货在途天数。调大→路上卖得更多、缺口更大、要发更多。                             |
