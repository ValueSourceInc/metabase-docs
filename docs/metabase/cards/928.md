## Description

全摊薄已售经营表现底表（按 Marketplace 隔离版）。在 #904 order_level 数据基础上，将 uncovered_platform_cost（无法归属到订单/SKU 的平台级费用）按 marketplace 分别分摊到该 marketplace 内的 SKU。 分摊基准（8 类平台费全覆盖，全池守恒）： - Storage → #806 库存量×单件体积，日摊分键 product_sales_usd - Inbound Placement → #806 入库运输数量，日摊分键 gross_sold_qty - Removal/Disposal → #733 损坏/缺陷退货数量（CUSTOMER_DAMAGED/CARRIER_DAMAGED/DEFECTIVE），日摊分键 cost_recognized_qty - Subscription / Promotion Deal / Vine / FBA fees / Selling fees → 按 product_sales_usd 分摊（无物理驱动量） - UNKNOWN marketplace 的费用按各 marketplace 销售额比例重新分配到 US/CA 列含义（重要）： - order_level_net_profit_usd = 订单级净利（销售额 - 订单级费用 - 广告 - 产品成本 - 物流成本），不含未归属平台费。等同于 #904 的 net_profit_usd。 - fully_loaded_net_profit_usd = order_level_net_profit_usd + 已分摊的 8 类平台费 = 全摊薄净利（SKU 已承担平台 overhead）。 注意：order_level_net_profit_usd 不含任何平台 overhead，会高估单 SKU 真实利润；看全摊薄口径用 fully_loaded_net_profit_usd。 已知限制： 1. #806 为当前库存快照，非历史时序 2. Subscription/Vine/Promo Deal/FBA/Selling 无物理驱动因素，按销售额分摊 3. 仅包含 order_level 数据（有 SKU），不含 uncovered_platform_cost 原始行 4. 分摊仅在 marketplace 维度隔离，custom_store 维度暂未隔离 5. 日摊分键为 product_sales_usd 的费用项：当某 SKU 累计销售额为 0 时其在 SKU 级分到的份额无法摊到天会被清零（storage 约 3-4% 丢失）。趋势分析可接受，精确对账需注意。 2026-06-25 fix1: #806 的 per-unit-volume 在 CA marketplace 全量为异常值（>100），AVG 时用 CASE 过滤 >10 的异常值（自动忽略 NULL，跨 marketplace 取同 SKU 真实体积），避免 CA storage 被脏数据扭曲。US 不受影响。 2026-06-26 fix2: 将 uncovered 的 fba_fees + selling_fees 也纳入分摊（原仅分摊 6 类，漏掉约 $1674），现 8 类全池守恒。 2026-06-26 fix3: 列重命名 net_profit_usd→order_level_net_profit_usd、adjusted_net_profit_usd→fully_loaded_net_profit_usd，明确'订单级净利(不含平台费)'与'全摊薄净利(含平台费)'的区别。 2026-06-29 fix4: 新增 Global Inbound Transportation Fees 分摊（原完全丢失，源端 -7125.85，sku 全空无法行级归集）。纳入 uncovered 池按 marketplace 分摊。 分摊权重调整（A方案）：Inbound Placement + Global Inbound Transportation 共用入仓体积权重 = afn-inbound-shipped-quantity × per-unit-volume（对齐两项费用的物理驱动：placement per-unit费率随size tier、transport按体积重），体积缺失退件数兜底。原 inbound_placement 用纯件数权重会扭曲大/小件分摊，现改体积权重后总额仍守恒(-13142.42)但SKU间分布更准。需同步改 #904（platform_costs白名单+列）与 #933（展示层加 Global Inbound Transportation Fees 行）。
